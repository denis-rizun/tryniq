# Backend Data Layer

Rules for Pydantic schemas, SQLModel tables, and configuration.

## Schemas (Pydantic)

- Inherit from `app.core.base_schema.BaseSchema` (sets `from_attributes=True`). Update payloads inherit from `UpdateSchema` (which enforces "at least one field provided").
- One file per feature, named `schemas.py`. Group request/response/event types here.
- Naming: `<Noun>CreateRequest`, `<Noun>UpdateRequest`, `<Noun>Response`, `<Noun>Event`.
- Discriminated unions use `type X = Annotated[A | B | C, Field(discriminator="kind")]` and a sibling `TypeAdapter` named `<NAME>_ADAPTER`. Use the PEP 695 `type` keyword, not `TypeAlias`.
- Do not subclass `BaseModel` directly in feature code — go through `BaseSchema` (or `_BaseEvent` for wire formats).

## Models (SQLModel)

- One table per class. Always set:

  ```python
  __tablename__ = "snake_case"
  model_config = SQLModelConfig(extra="allow")
  ```

- Table names are **singular** (`"meeting"`, `"participant"`, `"graph_node"`) — never plural. Reserved words are fine; SQLAlchemy quotes them, but raw SQL in migrations must quote them explicitly.
- Do not pass `nullable=False` — SQLModel infers nullability from the annotation (`str` → NOT NULL, `str | None` → NULL). Pass `nullable=` only to override that inference.
- Use the mixins in `app/core/database.py`: `IDMixin` (UUID PK with `uuid4` default) and `TimestampMixin` (created_at / updated_at, timezone-aware, with auto-update event listener). MRO: `class Foo(IDMixin, TimestampMixin, SQLModel, table=True): ...`.
- Datetimes are timezone-aware UTC: `Field(default_factory=lambda: datetime.now(tz=UTC), sa_type=DateTime(timezone=True))`. Never use naive datetimes.
- Foreign keys: `Field(foreign_key="<table>.id", index=True)`. JSONB columns: `Field(sa_column=Column(JSONB, nullable=True))`.
- Relationships default to `lazy="raise"`. Loading is explicit at the query site.
- Schema migrations live in `alembic/`. Never edit a committed migration; create a new one.

## Migrations (Alembic)

- **File and revision IDs are sequential four-digit numerics**, zero-padded — `0001`, `0002`, ..., `0010`, `0011`. Filename is `NNNN_<snake_case_description>.py`; the `revision = "NNNN"` and `down_revision = "<prev>"` strings match. `alembic revision --autogenerate` emits a random hash slug — **rename the file and edit the `revision`/`down_revision` strings** to the next sequential number before committing.
- Hand-clean every autogenerated migration: drop the `# ### commands auto generated by Alembic ###` comment markers (no-comments rule), confirm the `sqlmodel.sql.sqltypes` import is present if `AutoString` is referenced, and run `uv run ruff format` on the file.
- After hand-editing, run `uv run alembic check` against a DB at the previous head — it must report `No new upgrade operations detected`. Then verify the round-trip: `alembic upgrade head` → `downgrade -1` → `upgrade head`.
- **Postgres ENUM types must be created exactly once** by the migration that introduces them, and reused by `create_type=False` everywhere else. Define the ENUM at module scope, call `.create(..., checkfirst=True)` in `upgrade()` only in the owning migration, and `.drop(..., checkfirst=True)` in its `downgrade()`. Downstream migrations reference the same name with `create_type=False` and **must not** call `.create()` or `.drop()` — that would re-create or destroy a type other tables still depend on.

  Owning migration (creates the type):

  ```python
  from sqlalchemy.dialects.postgresql import ENUM

  meeting_status = ENUM(
      "LIVE",
      "FINALIZING",
      "FINAL",
      "FAILED",
      name="meetingstatus",
      create_type=False,
  )


  def upgrade() -> None:
      bind = op.get_bind()
      meeting_status.create(bind, checkfirst=True)
      op.create_table("meeting", sa.Column("status", meeting_status, nullable=False), ...)


  def downgrade() -> None:
      op.drop_table("meeting")
      meeting_status.drop(op.get_bind(), checkfirst=True)
  ```

  Downstream migration (reuses the type — no create/drop):

  ```python
  meeting_status = ENUM(..., name="meetingstatus", create_type=False)


  def upgrade() -> None:
      op.create_table("meeting_status_history", sa.Column("status", meeting_status, nullable=False), ...)


  def downgrade() -> None:
      op.drop_table("meeting_status_history")
  ```

## Configuration

- All settings flow through `pydantic-settings`. Each feature owns a `<Feature>Settings(BaseSettings)` class in `<feature>/config.py` with:

  ```python
  model_config = SettingsConfigDict(**BASE_MODEL_CONFIG, env_prefix="<FEATURE>_")
  ```

  using `BASE_MODEL_CONFIG` from `app/core/config.py`.
- The root `Settings` aggregator lives in `app/config.py` and is exposed as `config = Settings.get_instance()` (`@lru_cache`-cached). All code reads `from app.config import config` and accesses `config.<feature>.*`.
- Field names are **UPPER_SNAKE_CASE** inside settings classes (matches env var names). Sub-block attribute names on the root `Settings` are **lowercase**.
- Secrets use `pydantic.SecretStr`. Read with `.get_secret_value()` only at the point of use.
- Do **not** read `os.environ` directly. Add a setting instead.
- Email-typed fields (settings, schemas, ORM columns) use `pydantic.EmailStr`, never plain `str`. This includes optional fields (`EmailStr | None`) and bootstrap/admin settings — the validator runs at load time and fails fast on malformed env input.
